Introduction

Motivation for this project

Technical skills are fundamental to being a good professional, but knowing how to apply them productively is what really matters. It's like having an excellent set of tools and knowing the name of each one, but not being able to use them to build something concrete and meaningful.

That's how I felt. I have technical knowledge in programming, databases, data structures, Python, and more... but I still felt inexperienced. The applications I built seemed simple and amateurish, even when I tried my best. They were far below the solutions I saw from other professionals. Recognizing this gap was the first step.

While researching how to improve, I realized I needed to specialize before trying to do everything at once. For someone starting out, having clear direction makes all the difference. During this process, I came across the book "Lean Analytics: Use Data to Build a Better Startup Faster" by Alistair Croll and Benjamin Yoskovitz, which inspired me and reminded me of Cohort Analysis from my college statistics course—a topic that always interested me. I've always enjoyed mathematics, especially statistics and financial mathematics.

That's how this project was born: a way to work on strategic analysis, exercise my ability to connect data to business decisions, and above all, develop robust analysis with clear communication.

This material certainly has imperfections, but I'll use it to study and continually improve. It's my first draft and starting point for future projects.

Chosen technical stack and why

BigQuery:

I chose BigQuery as my data warehouse because of its ability to process SQL queries on large volumes without infrastructure setup. Additionally, the Free Tier offers 10GB of storage and 1TB of monthly processing, more than enough for this project. Queries that would take minutes in local PostgreSQL run in seconds on BigQuery thanks to its columnar architecture and distributed processing. I also wanted to learn a cloud service beyond AWS.

dbt Core:

dbt brings software engineering best practices to SQL: versioning, automated testing, auto-generated documentation, and code modularization. Instead of scattered SQL scripts, I have a well-documented pipeline. However, during the project I realized I didn't need dbt as much as I thought. I overestimated the complexity of my project, which is something difficult to learn from isolated exercises alone. It was a valuable lesson.

Looker Studio:

I'm an advanced Power BI user and intermediate with Tableau, but I wanted to try a different BI tool—that's where Looker Studio comes in. I chose it for its native integration with BigQuery and much lower learning curve compared to Power BI and Tableau.

Why I didn't use local PostgreSQL:

I love working with PostgreSQL, but it would require managing servers, backups, and scalability. I wanted to simulate a professional environment with serverless cloud solutions like BigQuery to reduce operational overhead. This project attempts to simulate real production architecture, albeit on a smaller scale.

What I learned from this project

  • How to structure a data warehouse in layers (staging → marts)
  • Implement cohort analysis to measure retention
  • Customer segmentation with RFM Analysis
  • Document data pipelines professionally
  • Free Tier limitations and how to work around them
  • Better tool sizing and selection
